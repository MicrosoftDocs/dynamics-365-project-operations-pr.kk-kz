# <a name="upgrade-from-project-service-automation-to-project-operations"></a>Project Service Automation нұсқасынан Project Operations нұсқасына жаңарту

Жаңарту үшін үш кезеңнің екіншісін хабарлауға қуаныштымыз Microsoft Dynamics 365 Project Service Automation Microsoft-қа Dynamics 365 Project Operations. Бұл мақала осы қызықты саяхатқа аттанатын тұтынушыларға шолу жасайды. 

Жаңарту бағдарламасы үш кезеңге бөлінеді.

| Жеткізуді жаңарту | 1 кезең (2022 ж. қаңтар) | 2 кезең (2022 ж. қараша) | 3 кезең (2023 жылғы сәуір толқыны)  |
|------------------|------------------------|---------------------------|---------------------------|
| Жобалар үшін жұмысты бөлу құрылымына (WBS) тәуелділік жоқ | :heavy_check_mark: | :heavy_check_mark: | :heavy_check_mark: |
| Жоба операцияларының қазіргі уақытта қолдау көрсетілетін шектеріндегі WBS | | :heavy_check_mark: | :heavy_check_mark: |
| Жобаның жұмыс үстелі клиентіне қолдауды қоса алғанда, Жоба операцияларының қазіргі уақытта қолдау көрсетілетін шектеулерінен тыс WBS | | | :heavy_check_mark: |

## <a name="upgrade-process-features"></a>Процесс мүмкіндіктерін жаңарту 

Жаңарту процесінің бөлігі ретінде әкімшілерге қателерді оңай диагностикалау мүмкіндігін беру үшін торап картасына жаңарту журналдарын қостық. Жаңа интерфейске қоса, жаңартудан кейін деректер тұтастығын қамтамасыз ету үшін жаңа тексеру ережелері қосылады. Жаңарту процесіне келесі тексерулер қосылады.

| Тексерулер | 1 кезең (2022 ж. қаңтар) | 2 кезең (2022 ж. қараша) | 3-кезең  |
|-------------|------------------------|---------------------------|---------------------------|
| WBS жалпы деректер тұтастығын бұзуға қарсы тексеріледі (мысалы, бір негізгі тапсырмамен байланыстырылған, бірақ әр түрлі негізгі жобалары бар ресурс тағайындаулары). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS сәйкес расталады [Web үшін жобаның белгілі шектеулері](/project-for-the-web/project-for-the-web-limits-and-boundaries). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS Project жұмыс үстелі клиентінің белгілі шектеулеріне қарсы тексеріледі. | |  | :heavy_check_mark: |
| Тапсырылатын ресурстар мен жоба күнтізбелері жалпы үйлеспейтін күнтізбе ережесінің ерекше жағдайларына қарсы бағаланады. | | :heavy_check_mark: | :heavy_check_mark: |

2-кезеңде Project Operations бағдарламасына жаңартқан тұтынушылар өздерінің бар жобаларын жобаны жоспарлау үшін тек оқуға арналған тәжірибеге дейін жаңартады. Бұл тек оқуға арналған тәжірибеде толық WBS бақылау торында көрінеді. WBS өңдеу үшін жоба менеджерлері таңдай алады [**Түрлендіру**](/PSA-Upgrade-Project-Conversion.md) жобаның негізгі бетінде. Фондық процесс жобаны жаңартады, осылайша ол Вебке арналған Project бағдарламасынан жаңа жобаны жоспарлау тәжірибесіне қолдау көрсетеді. Бұл кезең ішінде сәйкес келетін жобалары бар тұтынушылар үшін қолайлы [Web үшін жобаның белгілі шектеулері](/project-for-the-web/project-for-the-web-limits-and-boundaries).

3-кезеңде жобаларын сол қолданбадан өңдеуді жалғастырғысы келетін тұтынушылардың пайдасы үшін Project жұмыс үстелі клиентіне қолдау қосылады. Дегенмен, егер бар жобалар Веб тәжірибесіне арналған жаңа жобаға түрлендірілсе, қондырмаға кіру әрбір түрлендірілген жоба үшін өшіріледі.

## <a name="prerequisites"></a>Алғышарттар

1-кезеңді жаңартуға жарамды болу үшін келесі критерийлерге сай болуыңыз керек:

- Мақсатты ортада ешбір жазба болмауы керек **msdyn_projecttask** нысан.
- Жарамды Project Operations лицензиялары барлық белсенді пайдаланушыларға тағайындалуы керек. 
- Өндірістік ортаңызға сәйкес келетін өкіл деректер жиынын қамтитын кем дегенде бір өндірістік емес ортада жаңарту процесін растауыңыз керек.
- Мақсатты орта Project Service Automation Update Release 37 (V3.10.58.120) немесе одан кейінгі нұсқасына жаңартылуы керек.

2-кезеңді жаңартуға жарамды болу үшін келесі критерийлерге сай болуыңыз керек:

- Жарамды Project Operations лицензиялары барлық белсенді пайдаланушыларға тағайындалуы керек. 
- Өндірістік ортаңызға сәйкес келетін өкіл деректер жиынын қамтитын кем дегенде бір өндірістік емес ортада жаңарту процесін растауыңыз керек.
- Мақсатты орта Project Service Automation Update Release 37 (V3.10.58.120) немесе одан кейінгі нұсқасына жаңартылуы керек.
- Тапсырмаларды қамтитын орталарға (msdyn_projecttask) бір жобадағы тапсырмалардың жалпы саны 500 немесе одан аз болған жағдайда ғана қолдау көрсетіледі.

3-кезеңнің алғышарттары жалпы қолжетімділік күні жақындаған сайын жаңартылады.

## <a name="licensing"></a>Лицензиялау

Project Service Automation үшін белсенді лицензияларыңыз болса, Project Service Automation және т.б. барлық мүмкіндіктерін қамтитын Project Operations орнатуға және пайдалануға болады. Осылайша, сіз өндірісте Project Service Automation пайдалануды жалғастыра отырып, Project Operations мүмкіндіктерін тексере аласыз. Project Service Automation лицензияларының мерзімі аяқталғаннан кейін, Project Operations бағдарламасына өтуіңіз керек. Бұл ауысуды жоспарлағанда, Project Operations лицензиясында Project Service Automation лицензиясының жоқтығын ескеру қажет. Project Service Automation қолданбасын қолданған сценарийлері бар және олар Project Operations қызметіне көшуді жоспарлаған кезде PSA лицензияларын пайдалануды жалғастыру немесе ұлғайту қажет тұтынушылар Project Operations сатып алынған лицензияларға негізделген уақытша PSA лицензияларын сұрай алады. Бір Project Services лицензиясы үшін бір Project Service Automation лицензиясы беріледі. Уақытша PSA лицензияларын мына сілтеме арқылы сұрауға болады: aka.ms/ineedpsa

## <a name="testing-and-refactoring-customizations"></a>Теңшеулерді тестілеу және рефакторинг

Бастапқы нүкте ретінде импорттың сәтті болғанын және бизнес операцияларының күткендей әрекет ететінін растау үшін таза Project Operations (Lite) ортасына барлық теңшеулерді импорттаңыз.

Мұнда назар аудару керек нәрселер бар:

- Тәуелділіктер жоқ болғандықтан импорттау сәтсіз болуы мүмкін. Басқаша айтқанда, жоба әрекеттерінде жойылған теңшелімдерге сілтеме өрістері немесе басқа құрамдас бөліктер. Бұл жағдайда осы тәуелділіктерді әзірлеу ортасынан алып тастаңыз.
- Басқарылмайтын және басқарылатын шешімдеріңіз реттелмеген құрамдастарды қамтыса, сол құрамдастарды шешімнен жойыңыз. Мысалы, теңшеу кезінде **Жоба** нысан үшін шешіміңізге тек нысан тақырыбын қосыңыз. Барлық өрістерді қоспаңыз. Барлық ішкі құрамдастарды бұрын қосқан болсаңыз, жаңа шешімді қолмен жасап, оған сәйкес құрамдастарды қосуыңыз қажет болуы мүмкін.
- Пішіндер мен көріністер күткендей көрінбеуі мүмкін. Кейбір жағдайларда, қораптан тыс пішіндердің немесе көріністердің кез келгенін теңшеген болсаңыз, теңшеулер Жоба әрекеттеріндегі жаңа жаңартулардың күшіне енуіне жол бермеуі мүмкін. Бұл мәселелерді анықтау үшін жоба операцияларының таза орнатылымын және теңшеулерді қамтитын Жоба операцияларының орнатылымын қатар қарап шығуды ұсынамыз. Пішін нұсқаңыздың әлі де мағыналы екенін және пішіннің ештеңе жазылмаған нұсқасынан бірдеңені жоғалтпағанын растау үшін бизнесіңізде жиі қолданылатын пішіндерді салыстырыңыз. Өзіңіз реттеген кез келген көріністер үшін қатарлас шолудың бірдей түрін орындаңыз.
- Іскерлік логика орындалу уақытында сәтсіз болуы мүмкін. Қосылатын модульдердегі өрістерге сілтемелер импорттау кезінде тексерілмегендіктен, бизнес логикасы енді жоқ өрістерге сілтемелерге байланысты сәтсіз болуы мүмкін және келесі мысалға ұқсайтын қате туралы хабарды алуыңыз мүмкін: "'Жоба' нысанда Name = 'msdyn_plannedhours' және NameMapping = 'Логикалық' атрибуттары жоқ." Бұл жағдайда теңшеулерді жаңа өрістерді пайдаланатындай өзгертіңіз. Қосылатын модуль логикасында автоматты түрде жасалған прокси сыныптарын және күшті түр сілтемелерін пайдалансаңыз, сол прокси-серверлерді таза орнатудан қайта жасауды қарастырыңыз. Осылайша, қосылатын модульдер ескірген өрістерге тәуелді болатын барлық орындарды оңай анықтауға болады.

Жоба операцияларын таза импорттау үшін теңшеулерді жаңартқаннан кейін келесі қадамдарға өтіңіз.

## <a name="end-to-end-testing-in-development-environments"></a>Әзірлеу орталарында түпкілікті тестілеу

### <a name="initiate-upgrade"></a>Жаңартуды бастаңыз 

1. Ішінде Power Platform басқару орталығына өтіңіз, ортаңызды тауып, таңдаңыз. Содан кейін қолданбалардан тауып, таңдаңыз **Dynamics 365 Project Operations**.
2. таңдаңыз **Орнату** жаңартуды бастау үшін. The Power Platform басқару орталығы бұл орнатуды жаңа орнату ретінде ұсынады. Дегенмен, Project Service Automation бағдарламасының бұрынғы нұсқасының болуы анықталады және бар орнату жаңартылады.

    Жаңарту аяқталғаннан кейін орта Project Operations орнатылғанын және Project Service Automation орнатылмағанын көрсетуі керек.

    Ортадағы деректер көлеміне байланысты жаңарту бірнеше сағатқа созылуы мүмкін. Жаңартуды басқаратын негізгі топ сәйкесінше жоспарлап, жаңартуды жұмыстан тыс уақытта іске қосуы керек. Кейбір жағдайларда, деректер көлемі үлкен болса, жаңарту демалыс күндері іске қосылуы керек. Жоспарлау туралы шешім төменгі ортадағы сынақ нәтижелеріне негізделуі керек.

3. Сәйкесінше реттелетін шешімдерді жаңартыңыз. Осы кезде теңшеулеріңізге жасаған кез келген өзгертулерді файлда орналастырыңыз [Теңшеулерді тестілеу және рефакторинг](#testing-and-refactoring-customizations) осы мақаланың бөлімі.
4. Бару **Параметрлер** \> **Шешімдер**, және жою үшін таңдаңыз **Жоба операциялары ескірген құрамдас бөліктер** шешім.

    Бұл шешім бар деректер үлгісін және жаңарту кезінде бар құрамдастарды сақтайтын уақытша шешім болып табылады. Бұл шешімді жою арқылы сіз енді пайдаланылмайтын барлық өрістер мен құрамдастарды жоясыз. Осылайша сіз интерфейсті жеңілдетуге және интеграция мен кеңейтуді жеңілдетуге көмектесесіз.
    
### <a name="upgrade-to-project-operations-lite"></a>Project Operations Lite нұсқасына жаңартыңыз

Келесі қадамдар жаңарту процесін және байланысты қателерді тіркеуді сипаттайды:

1. **PSA нұсқасын тексеру:** Project Operations орнату үшін сізде V3.10.58.120 немесе одан жоғары болуы керек.
1. **Алдын ала тексеру:** Әкімші жаңартуды бастаған кезде, жүйе Жоба операциялары шешіміне негізгі болып табылатын әрбір нысанда алдын ала тексеру әрекетін іске қосады. Бұл қадам барлық нысандар сілтемелерінің жарамдылығын тексереді және WBS-ке қатысты деректердің Веб жобасының жарияланған шектеулерінде болуын қамтамасыз етеді.
1. **Метадеректерді жаңарту:** Сәтті алдын ала тексеруден кейін жүйе схемаға өзгертулерді бастайды және ескірген құрамдастардың шешімін жасайды. Теңшеулердің барлық қажетті рефакторингін аяқтағаннан кейін бұл ескірген шешімді жоюға болады. Бұл қадам жаңарту процесінің ең ұзақ бөлігі болып табылады және оны аяқтау төрт сағатқа дейін созылуы мүмкін.
1. **Деректерді жаңарту:** Барлық қажетті схема өзгерістері метадеректерді жаңарту қадамында аяқталғаннан кейін деректеріңіз жаңа схемаға тасымалданады және кез келген қажетті әдепкі және қайта есептеулер орындалады.
1. **Жоба кестесінің қозғалтқышын жаңарту:** Деректерді сәтті жаңартудан кейін **Кесте** негізгі беттегі қойындысы қайта белгіленген **Тапсырмалар**. Пайдаланушы жаңартудан кейін осы қойындыны таңдағанда, олар WBS тек оқуға арналған нұсқасын көру үшін бақылау торына өтуге бағытталады. WBS өңдеу үшін олар кестені бастауы керек [түрлендіру процесі](/PSA-Upgrade-Project-Conversion.md). Бұрыннан бар WBS жоқ барлық жобалар жаңа жоспарлау тәжірибесін түрлендірусіз тікелей пайдалана алады.
 
### <a name="validate-common-scenarios"></a>Жалпы сценарийлерді растаңыз

Арнайы теңшеулерді растаған кезде, қолданбаларда қолдау көрсетілетін бизнес процестерді де қарап шығуды ұсынамыз. Бұл бизнес-процестерге баға ұсыныстары мен келісім-шарттар сияқты сату нысандарын құру, сондай-ақ WBS-ті қамтитын жобаларды жасау және нақты деректерді бекіту кіреді, бірақ олармен шектелмейді.

## <a name="major-changes-between-project-service-automation-and-project-operations"></a>Project Service Automation және Project Operations арасындағы негізгі өзгерістер

Бұл бөлім Project Service Automation және Project Operations арасында күтуге болатын негізгі өзгерістердің қысқаша мазмұнын береді.

### <a name="project-planning"></a>Жобаны жоспарлау

Жоба операцияларындағы жобаны жоспарлау мүмкіндіктері енді клиенттік логика мен серверлік логика тіркесіміне сүйенбейді. Оның орнына Project Operations бағдарламасы Вебке арналған жобаны жоспарлау механизмі ретінде пайдаланады. Жоспарлау мүмкіндіктерін өзгерту бірнеше жаңа мүмкіндіктерге мүмкіндік береді, мысалы, кеңес және Гант көріністері, ресурстарға негізделген жоспарлау, [тапсырмаларды тексеру тізімінің элементтері](https://support.microsoft.com/office/use-task-checklists-in-microsoft-project-for-the-web-c69bcf73-5c75-4ad3-9893-6d6f92360e9c), және жобаны жоспарлау режимдері. Жаңа жоспарлау мүмкіндіктеріне жаңаның бай жиынтығы да қолдау көрсетеді [қолданбалы бағдарламалау интерфейстері (API)](../project-management/schedule-api-preview.md). Бұл API интерфейстері WBS жүйесіндегі нысанды жасауға, жаңартуға немесе жоюға арналған ешқандай бағдарламалық операция кестедегі есептелген өрістерді бұзбауын қамтамасыз етуге көмектесуге арналған.

## <a name="frequently-asked-questions"></a>Жиі қойылатын сұрақтар

### <a name="which-deployment-types-are-currently-supported-for-upgrade"></a>Қазіргі уақытта жаңарту үшін қандай орналастыру түрлеріне қолдау көрсетіледі?

### <a name="how-can-i-install-project-operations-before-the-upgrade-tooling-is-available"></a>Жаңарту құралдары қол жетімді болмай тұрып, Жоба операцияларын қалай орнатуға болады?

Жаңарту құралы қолжетімді болғанға дейін Project Operations орнатудың екі нұсқасы бар:

- Жаңа ортаны қамтамасыз ету.
- Project Service Automation жоқ кез келген сату ұйымында Жоба операцияларын бөлек орналастырыңыз.

Project Service Automation ұйымда орнатылған болса, бірақ ол пайдаланылмаған болса, оны жоюға болады. Project Service Automation толық жойылғаннан кейін, Project Operations бір ұйымға орнатылуы мүмкін.
